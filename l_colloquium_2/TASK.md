1. Как устроен Dockerfile и в чём разница между CMD и ENTRYPOINT?

> Там лежит инструкция по сборке образа, из которого запускается контейнер, настраиваются порты, глобальное окружения, какие файлы запустить при инициализации
> и тд., ну буквально - конфигурация контейнера; CMD - это набор команд которые выполняться при запуске по умолчанию, но их можно изменить,
> а ENTRYPOINT - набор команд который никак нельзя изменить и они гарантированно выполняться при запуске контейнера.

2. Для чего используется Docker Compose и как устроен его конфиг?

> Для того чтобы поднимать одновременно несколько контейнеров одного проекта, например, бэк и фронт. В его конфиге просто настраивается
> название контейнера, порт на котором он слушает, откуда собирать образ, его окружение (глобальные переменные), порядок запуска контейнеров,
> volume и прочее.

3. Как используется OpenAPI (Swagger) на практике?

> Это формальное описание REST API: какие есть endpoints, HTTP-методы, параметры, тела запросов, ответы, коды статусов и схемы DTO,
> позволяет избегать недопониманий между различными командами.

4. Как работают ORM, JPA и Hibernate и как задаются связи?

> ORM - это подход, который связывает объекты в коде с таблицами в БД. JPA - спецификация Java для ORM, а Hibernate - её реализация.
> Сущности помечаются @Entity, поля связываются с колонками, а связи задаются аннотациями @OneToOne, @OneToMany, @ManyToOne, @ManyToMany.
> Для внешнего ключа обычно используют @JoinColumn, а для связи many-to-many - отдельную таблицу через @JoinTable.

5. Почему возникает проблема N+1?

> Особенность ORM, возникает из-за lazy loading. Если мы запросим у БД книги по авторам, то выполнится сначала запрос на авторов, потом для каждого автора
> выполнится еще запрос на поиск его книг, таким образом мы совершим 1+N запросов.

6. Какие части входят в HTTP-запрос и HTTP-ответ?

> HTTP-запрос включает метод, URL, query-параметры, headers и body.
> 
> HTTP-ответ включает status code, headers и body.

7. Чем отличаются REST, gRPC и SOAP?

> REST - популярная спецификация API на HTTP, работает с JSON; gRPC - вызывает удалённые методы по строгому контракту .proto, может вызывать метод который
> физически находится в другом приложении, очень быстрый, часто используется в микросервисной архитектуре;
> SOAP - устаревший, медленный и неактуальный, базируется на XML.

8. Как настроить приложение через application.yaml и как читать значения оттуда?

> Главный файл конфигурации Spring-приложения, там указывается порт, база данных и другие особенности, например, логирование.
> Читать через аннотацию @Value (одно значение) или @ConfigurationProperties (несколько настроек).

9. Как устроено логирование в Spring и какие уровни используются?

> В Sping это архитектурная связка SLF4J (набор методов, интерфейс) и Logback (реализация), но мы сами решаем что, где и как логировать.
> Уровни: trace (вся инфа), debug (отладочная инфа), info (важные события), warn (предкпреждения: подозрительные моменты), error (ошибки).

10. Какие плюсы и минусы у микросервисов?

> Плюсы: независимый деплой каждого сервиса, возможность масштабировать каждый сервис отдельно, независимость сервисов (если один упадёт -
> остальные продолжат работать), возможность использовать разный стек технологий для каждого сервиса, возможность разрабатывать каждый сервис независимо.
> 
> Минусы: затратнее по вычислительным ресурсам, сложный дебаг, сложное тестирование, необходим полный мониторинг всех микросервисов одновременно (затратно и сложно).

10. Какие плюсы и минусы у монолитов? 

> Плюсы: проще писать, проще запускать, проще тестировать, дешевле развёртывать, меньше инфраструктуры, всё в одном месте
> 
> Минусы: даже для самого маленького изменения необходим деплой всего монолита, если что-то упадёт - весь сервис упадёт, масштабируемость очень ограничена
> (нельзя масштабировать что-то конкретное, только весь монолит целиком), один стек технологий, неудобно разрабатывать при большом кол-ве команд

11. Как CRUD соотносится с HTTP-методами и кодами ответов?

> AC: 200 - OK, 201 - Created, 204 - No Content, 400 - Bad Request, 401 - Unauthorised, 403 - Forbidden, 404 - Not Found, 409 - Conflict,
> 429 - Too many Request, 500 - Internal Server Error, 502 - Bad Gateway, 504 - Gateway Timeout и другие
> 
> CRUD-HTTP: CREATE - POST, READ - GET, UPDATE - PUT/PATCH, DELETE - DELETE

12. В чём разница между аутентификацией и авторизацией и как используется JWT?

> Аутентификация - проверка кто ты (ввёл логин пароль), авторизация - проверка что тебе можно. JWT - подписанный токен после аутентификации
> между пользователем и сервером: при каждом запросе клиент также отправляет этот токен, а сервер проверяет его подлинность.
> У пользователя обычно хранится в Secured Cookies.

13. Какие задачи решает Docker?

> Вся программа разворачивается в контейнере который одинаков для всех типов устройств/окружений/машин. То есть не возникает проблемы что
> у меня на компе всё работает, а у друга оно лежит из-за разных ОС.

14. Как используется @RestController и связанные аннотации?

> Используется для создания REST API, принимает HTTP запросы, вызывает нужные методы и возвращает данные. @RequestMapping - путь; @GetMapping - обработка GET;
> @PostMapping - обработка POST; @PutMapping - обработка PUT и тд;; RequestBody - читает тело запроса, а PathVariable достаёт значение из url

15. Чем отличаются unit, integration и E2E тесты?

> Unit тесты самые дешёвые и пишутся разработчиком сразу чтобы протестировать их фичу в коде, проверяют маленький изолированный кусок;
> integration тесты проверяют совместную работу нескольких модулей,
> e2e тесты очень дорогие и долгие, проверяют полный пайплайн системы, как реальный пользователь

> Очев.

17. Зачем нормализуют базы данных?

> Чтобы убрать дублирование данных, уменьшить риск ошибок и хранить связанные данные в отдельных таблицах.
> 
> Нормальная форма БД: 1 значение в 1 ячейке, все поля зависят от PK, поля не должны зависеть друг от друга

18. Как организовать unit-тесты с JUnit и MockK?

> Выбираем один небольшой кусок логики, обычно один класс и тестируем его изолированно. Его зависимости не используем настоящие, а заменяем mock-объектами
> через MockK. JUnit нужен для структуры теста: @Test, проверки через assertEquals, assertThrows и т.д. MockK нужен, чтобы подменять зависимости.

19. Как используются логи и метрики в приложении?

> Мониторинг, дебаг и поиск ошибок.

20. Что такое Continuous Integration (CI)?

> Это автоматическая проверка кода: при push/pull обычно собирается проект и запускаются определённые тесты / другие проверки.
> На GitHub это настраивается через GitHub Actions в workflow-файле, где ссодержится ин-фа что и при каком условии делать.
> Если тест выдал ошибку, то push/pull может быть отклонён репозиторием.

21. Как собрать и запустить контейнер через Dockerfile?

> Собрать ```docker build -t {NAME} .```, запустить ```docker run -p 8080:8080 {NAME}```

22. Как Spring находит и создаёт бины?

> При запуске проекта сканирует его и находит свои аннотации (@Component, @Service, @Repository, @Controller, @RestController), создаёт для них объект и
> кладёт в свой контейнер (Spring Container).

23. Для чего нужна аннотация @Transactional?

> Чтобы несколько операций с БД выполнялись как одна транзакция и если одна из операций упала - отменить изменения.

> Через файлы application.yaml, переменные окружения (env), параметры запуска и смену профиля (prod, dev, test).

> Само логирование прописывается в коде, а настраивается в application.yaml (уровень, формат и тд.)

32. Как работают сущности и связи в JPA/Hibernate?

> Это классы, помеченные @Entity, которые отображаются на таблицы в реляционной БД, поля сущности соответствуют колонкам, а объект сущности соответствует
> строке таблицы. Hibernate управляет этими объектами и сам генерирует SQL-запросы для сохранения, обновления, удаления и получения данных.
> Связи между сущностями задаются аннотациями @OneToOne, @OneToMany, @ManyToOne, @ManyToMany, а внешние ключи обычно указываются через @JoinColumn.

33. Как применяется JWT в авторизации?

> JWT используется как токен доступа.

34. Как обнаружить и устранить проблему N+1?

> По логам Hibernate. Фиксится через JOIN, прямой SQL запрос в @Query или @EntityGraph.

35. Как устроена пирамида тестирования?

> Unit > Integration > E2E (дешёвый и быстрый > дорогой и долгий)

36. Как обрабатываются HTTP-запросы в Spring REST?

> В Spring REST HTTP-запрос сначала попадает в DispatcherServlet, потом Spring находит подходящий метод в @RestController, достаёт данные из
> URL/body/headers, вызывает метод контроллера, дальше обычно идёт Service → Repository → Database, а результат возвращается клиенту как HTTP-response.

37. Как используются JUnit 5 и MockK при тестировании?

> JUnit нужен для структуры теста: @Test, проверки через assertEquals, assertThrows и т.д. MockK нужен, чтобы подменять зависимости.

38. Как выбирать между Contract First и Code First?

> Contract First - большой проект, где несколько команд и уже есть договорённости по API
> 
> Code First - когда одна команда и небольшой проект

39. Что такое Continuous Delivery (CD)?

> Метод, при котором приложение после изменений в коде автоматически проходит сборку, тесты, создаётся готовый к деплою образ,
> деплой происходит автоматически или по запускается вручную.


40. Как мониторить приложение с помощью логов и метрик?

> Отслеживать ключевые метрики, смотреть в них аномалии. Читать warnings и errors в логах.

41. В чём особенности контейнеров Docker?

> Это изолированная среда, одинаковая для всех окружений, где разворачивается приложение.

42. В чём суть IoC и DI в Spring и как они упрощают разработку?

> IoC - это принцип, при котором не наш код вручную создаёт объекты и связывает их, а этим управляет Spring Container.
> 
> DI - это способ реализации IoC: Spring сам внедряет нужные зависимости в классы, чаще всего через конструктор.

43. Какую роль играют Spring бины и чем отличаются основные аннотации?

> Spring bean — это объект, который создаётся и контролируется Spring Container. Бины нужны, чтобы Spring мог сам связывать части приложения через DI.
> Основные аннотации @Component, @Service, @Repository, @Controller, @RestController создают beans, но обозначают разные роли классов:
> общий компонент, бизнес-логика, работа с БД, web-контроллер и REST-контроллер.
